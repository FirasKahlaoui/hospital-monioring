\section{System Overview}

The firmware maps the software architecture described in
Chapter~\ref{ch:architecture} directly onto the ESP32's dual-core execution model.
At boot, \texttt{setup()} initialises LittleFS, loads NVS configuration, initialises the sensors, starts
both Wi-Fi interfaces, registers HTTP routes, and launches the Firebase FreeRTOS task .
After \texttt{setup()} returns, the Arduino \texttt{loop()} continues to execute, calling \texttt{updateSensors()} on every iteration and scheduling periodic
Wi-Fi background scans.

The \textbf{Backend Subsystem} is implemented as a Node.js service that acts as a secure proxy between the clinical dashboard and the Firebase infrastructure. It consumes data from Firestore for administrative operations and manages the alert logic by evaluating incoming telemetry. The server is initialized via a tRPC router which binds the service logic to the network layer, ensuring that every API call is strictly validated against the project's TypeScript interfaces.

The \textbf{Web Dashboard} is a high-fidelity React application that serves as the central command center for hospital staff. It implements a reactive state model where UI components automatically re-render in response to Firebase Realtime Database events. This tight coupling between the cloud database and the frontend components allows for a "zero-refresh" monitoring experience, which is critical for observing acute physiological changes.

\section{Hardware Implementation}
% ============================================================

The hardware assembly consists of three components mounted on a common 3.3\,V power
rail:

\begin{itemize}
	\item \textbf{ESP32 development board}.
	\item \textbf{DHT22} connected to GPIO~4.
	\item \textbf{MAX30102 breakout board} connected to the ESP32's default
	      I\textsuperscript{2}C bus (SDA: GPIO~21, SCL: GPIO~22), powered at 3.3\,V.
\end{itemize}

All components are powered from the ESP32 development board's 3.3\,V regulated output.
The board is powered from USB 5\,V connected to a mains adapter in the hospital room.

\section{Software Implementation}
% ============================================================

\subsection{Embedded Subsystem}

\subsubsection{Firebase Initialisation on Core~0}

The \texttt{initFirebase()} function is called from inside
\texttt{firebaseUploadTask()}, which is pinned to Core~0 via
\texttt{xTaskCreatePinnedToCore(..., 0)}.

\subsubsection{Configuration Portal}

The configuration portal is served entirely from the LittleFS partition.
The \texttt{isAPConnected()} guard checks whether the incoming request arrived via
the AP subnet (\texttt{WiFi.softAPIP()}) and redirects any STA-side request to the
root path, preventing access from the hospital network.

\subsubsection{Wi-Fi Recovery}

All reconnection logic is consolidated in the \texttt{wifiConnect()} helper, which
always calls \texttt{WiFi.disconnect()} followed by
\texttt{WiFi.begin(ssid, password)}.

\subsection{Backend Implementation}
The backend implementation is modularized into \textbf{functional routers} (e.g., \texttt{people}, \texttt{patients}, \texttt{events}, \texttt{ai}). Each router defines its procedures using Zod for runtime schema validation. The \textbf{Alert Engine} utilizes a dedicated SMTP module configured with a Gmail relay to dispatch clinical notifications. Authentication is implemented as a middleware layer that decodes and verifies Firebase JWTs before granting access to protected clinical procedures.

\subsection{Web Dashboard Implementation}
The dashboard's core is the \textbf{Real-time Monitoring Component}, which uses a custom hook to manage the Firebase RTDB listener lifecycle. Vitals are visualized using \textbf{AreaCharts} from Recharts, optimized for high-frequency updates without UI stutter. A secondary \textbf{AI Insights Panel} renders synthesized clinical conclusions by subscribing to the tRPC AI results. The personnel tracking system is implemented via a \textbf{WebWorker} running \textbf{face-api.js} to ensure face detection does not block the main UI thread.

\section{Software Validation}
% ============================================================

\subsection{Embedded Subsystem Validation}

Validation was performed across three dimensions:

\textbf{Sensor output validation.}
DHT22 readings were cross-referenced against a calibrated reference thermometer.

\textbf{Firebase upload validation.}
The Firebase Realtime Database console was monitored in real time to confirm record
arrival, correct path structure (\texttt{/rooms/\{roomId\}} and
\texttt{/patients/\{patientId\}}), correct field names and data types, and
correct server-generated timestamps.

\textbf{Authentication validation.}
The serial monitor (debug build) was used to trace the complete authentication event
sequence: \textit{authenticating} $\to$ \textit{auth request sent} $\to$
\textit{auth response received} $\to$ \textit{token obtained}.
Error conditions (invalid API key, incorrect password) were deliberately induced
and confirmed to produce the expected HTTP~400 error codes via the callback.

\subsection{Backend Validation}
The backend was validated through a suite of \textbf{integration tests} using a tRPC caller. Test cases included:
\begin{itemize}
    \item \textbf{Auth Verification:} Confirming that requests without valid tokens return HTTP 401.
    \item \textbf{Schema Enforcement:} Ensuring that malformed JSON payloads are rejected by Zod validators.
    \item \textbf{Alert Dispatch:} Verifying that threshold violations trigger the SMTP relay and produce a "Success" status in the log.
\end{itemize}

\subsection{Web Dashboard Validation}
UI validation focused on \textbf{performance under load} and \textbf{clinical accuracy}.
\begin{itemize}
    \item \textbf{Stress Testing:} Verified that the dashboard maintains 60 FPS while rendering 10 concurrent vital sign charts.
    \item \textbf{Personnel Detection:} Confirmed that face recognition identifies authorized staff within <2 seconds with 90\%+ confidence.
    \item \textbf{Responsive Design:} Validated layout integrity across Desktop, Tablet, and Mobile viewport sizes using Chrome DevTools.
\end{itemize}

\section{Embedded Implementation (ESP32)}
% ============================================================

\subsection{Memory Optimisation}

The LittleFS image partition is mounted with \texttt{LittleFS.begin(true)}, enabling automatic
formatting on first boot.
NVS storage uses the \texttt{Preferences} library with the \texttt{"settings"}
namespace, consuming at most 10 string entries at typical credential lengths.

\subsection{Latency Optimisation}

Delta filtering (Equations~\ref{eq:room_thresh} and~\ref{eq:patient_thresh}) reduces
the frequency of HTTPS requests to only those carrying new information, keeping the
per-upload round-trip time as the dominant latency term rather than queuing delays.

\section{AI Integration}
The AI prediction layer is integrated as a \textbf{server-side service} that processes data on-demand from the dashboard. This deployment strategy was chosen to leverage the Node.js server's computational resources, keeping the ESP32 firmware lightweight.
\begin{itemize}
    \item \textbf{Integration:} The forecasting logic is wrapped in a tRPC query that accepts a 20-point telemetry window.
    \item \textbf{Optimization:} Z-score calculations and linear regressions are performed using optimized math functions to minimize inference latency ($<$10\,ms).
    \item \textbf{Validation:} The AI output was validated against synthetic "stress" datasets containing simulated cardiac events (tachycardia and SpO2 drops).
\end{itemize}
