\section{Functional Architecture}

The system is functionally decomposed into four independent subsystems, each with a
well-defined interface:

\begin{description}
	\item[Sensor Subsystem (ESP32, Core~1).]
	\item[Network/Firebase Subsystem (ESP32, Core~0).]
	\item[Settings/Web Server Subsystem (ESP32, Core~1).]
	\item[Configuration and Globals Layer.]
\end{description}

\subsection*{Backend Functional Decomposition}
The backend is architected as a \textbf{Modular tRPC Router} system. The \textbf{Authentication Middleware} intercepts every procedure call to verify Firebase ID tokens. The \textbf{Clinical Management Layer} handles Firestore operations for staff and patients, while the \textbf{Notification Engine} monitors the Realtime Database to trigger SMTP-based clinical alerts. Finally, the \textbf{AI Procedure Layer} provides windowed data analysis for predictive insights.

\subsection*{Web and AI Functional Decomposition}
The web dashboard is decomposed into three primary functional blocks: (1) the \textbf{Telemetry Subscription Bridge}, which maintains the high-frequency link to the Realtime Database; (2) the \textbf{Clinical Visualization Engine}, which manages the rendering of Recharts-based vitals; and (3) the \textbf{Local Intelligence Module}, which executes face-recognition models for personnel tracking. The AI subsystem decomposes into a \textbf{Statistical Signal Processor} for anomaly detection and a \textbf{Trend Extrapolator} for predictive forecasting.

\section{Hardware Architecture}

\begin{description}
	\item[Microcontroller.]
	      ESP32-WROOM-32, dual-core at 240\,MHz,
	      520\,KiB SRAM, 4\,MiB flash.

	\item[Temperature and Humidity Sensor.]
	      DHT22, single-bus digital interface on GPIO~4.

	\item[Pulse Oximeter and Heart Rate Sensor.]
	      MAX30102, I\textsuperscript{2}C fast mode (400\,kHz), SDA: GPIO~21,
	      SCL: GPIO~22.

	\item[Communication.]
	      IEEE 802.11b/g/n Wi-Fi radio integrated on-chip.
	      AP interface (192.168.4.1) serves the configuration portal.
\end{description}

\section{Cloud and Server Infrastructure}
The system is deployed on a \textbf{Serverless Cloud Infrastructure} to ensure high availability and scalability.
\begin{description}
	\item[Compute Tier.] The backend is hosted as a Node.js runtime environment on Vercel, utilizing global edge functions to minimize tRPC response times.
	\item[Persistence Tier.] Firebase Firestore serves as the document-based repository for clinical staff and patient records.
	\item[Streaming Tier.] Firebase Realtime Database provides the low-latency JSON bus for real-time sensor telemetry.
\end{description}

\section{Software Architecture}

\subsection*{Embedded Software (ESP32)}

The firmware is organised into six source files following a strict, acyclic include
hierarchy.
Table~\ref{tab:sw_modules} summarises the modules, their execution core, and their
responsibilities.

\begin{table}[H]
	\centering
	\caption{ESP32 firmware module summary.}
	\label{tab:sw_modules}
	\begin{tabularx}{\linewidth}{lllX}
		\toprule
		\textbf{File}           & \textbf{Context} & \textbf{Responsibility}                                                  \\
		\midrule
		\texttt{globals.h}      & Header           & Shared variable declarations, DBG macros, timing constants               \\
		\texttt{sensors.h/cpp}  & Loop             & DHT22 and MAX30102 acquisition, BPM and SpO\textsubscript{2} computation \\
		\texttt{network.h/cpp}  & FreeRTOS task    & Firebase auth, upload, Wi-Fi management                                  \\
		\texttt{settings.h/cpp} & ISR callbacks    & HTTP routes, LittleFS, NVS persistence                                   \\
		\texttt{esp32.ino}      & Entry point      & Global object definitions, \texttt{setup()}, \texttt{loop()}             \\
		\bottomrule
	\end{tabularx}
\end{table}

The dependency graph is strictly acyclic: \texttt{esp32.ino} includes all modules;
\texttt{network.h/cpp} and \texttt{settings.h/cpp} include \texttt{globals.h};
\texttt{sensors.h/cpp} includes \texttt{globals.h}.
No module includes another module's header directly.

\subsection*{Backend Software}
The service layer is implemented as a \textbf{Node.js} application using the \textbf{tRPC} framework to provide a strictly typed API. This architecture eliminates the need for manual schema synchronization between the server and client. The backend utilizes the \textbf{Firebase Admin SDK} for secure authentication verification and uses \textbf{Firestore} as the primary system of record for staff and patient data. Automated alert logic is processed in real-time by monitoring the telemetry stream against patient-specific thresholds.

\subsection*{Web Dashboard Software}
The user interface is a single-page application (SPA) built with \textbf{React} and \textbf{Vite}. It leverages the \textbf{Shadcn UI} component library for a modern, responsive clinical aesthetic. The application utilizes a reactive subscription model to the Firebase Realtime Database, ensuring that patient vitals are updated on-screen with sub-second latency. Data visualization is handled by the \textbf{Recharts} library, which renders interactive time-series plots of the patient's physiological trends.

\subsection*{AI Software}
The AI prediction layer is integrated as a set of tRPC procedures that process windowed telemetry vectors. The system employs \textbf{statistical anomaly detection} using Z-score analysis to identify clinical outliers. For forecasting, the engine applies a \textbf{linear regression model} to project vital sign trends for the subsequent 5-minute interval. This predictive capability allows the dashboard to provide "AI Insights" that warn clinicians of potential instability before standard threshold alerts are triggered.

\section{Processing Pipeline}

The end-to-end data processing pipeline for a single sensor reading, from hardware
to cloud record, proceeds as follows:

\begin{enumerate}
    \item \textbf{Acquisition:} The ESP32 Core 1 samples the MAX30102 and DHT22 sensors.
    \item \textbf{Processing:} Raw photoplethysmogram (PPG) data is processed via a peak-detection algorithm to derive heart rate and SpO2.
    \item \textbf{Persistence:} Core 0 authenticates with Firebase and pushes the JSON-serialized vitals to the Realtime Database.
    \item \textbf{Subscription:} The Web Dashboard, listening via WebSockets, receives the update and renders it in the live monitor.
    \item \textbf{Inference:} The Dashboard periodically sends a 20-point history to the tRPC AI endpoint for anomaly analysis and forecasting.
    \item \textbf{Alerting:} If the backend detects a threshold violation or a high-severity AI anomaly, an email is dispatched via the SMTP relay.
\end{enumerate}
