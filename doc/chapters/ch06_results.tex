\section{Experimental Protocol}

\subsection{Embedded Subsystem Protocol}

\begin{description}
	\item[Hardware setup.]
	      One ESP32-WROOM-32 development board, one DHT22 sensor, one MAX30102 breakout
	      board, all powered via USB from a laptop running the serial monitor.

	\item[Network environment.]
	      2.4\,GHz 802.11n Wi-Fi network.
	      Firebase Realtime Database.

	\item[Firmware build.]
	      Debug build (\texttt{deploy.sh --code --debug}) for all experiments, to capture
	      serial output.
\end{description}

\subsection{Backend Protocol}
The backend is validated using a \textbf{tRPC server-side caller} and
\textbf{Vitest} for unit testing. The test environment targets a local Node.js
runtime and optionally a Firestore emulator. Test cases cover authentication,
schema validation, and alert/notification procedures.

\subsection{Web Dashboard Protocol}
The dashboard should be tested across Chrome and Firefox on both Windows and Linux.
Real-time update latency can be measured using the \textbf{Chrome DevTools Performance profiler}. Test scenarios include multi-patient monitoring views (up to 5 concurrent streams) and automated face detection triggers for authorized and unauthorized personnel.

\subsection{AI Protocol}
The AI subsystem is evaluated using synthetic clinical datasets containing
simulated vital sign transitions (e.g., resting to tachycardia). Evaluation
metrics include \textbf{Mean Squared Error (MSE)} for forecasting accuracy and
\textbf{F1-score} for anomaly detection precision and recall. Tests are executed
on the Node.js service layer using the same sliding-window inputs as the
dashboard.

\section{Software Results (PC / Server)}

The repository includes automated tests for authentication logout, events logging,
alert creation, and patient CRUD flows (\texttt{alerts.test.ts},
\texttt{events.test.ts}, \texttt{patients.test.ts}). These tests validate the
shape of API responses and enforce consistent procedure behavior. Formal
performance benchmarks (latency, throughput) are not yet recorded and remain a
required measurement in the final validation campaign.

\subsection{Latency Analysis}
The end-to-end latency is dominated by the network round-trip time (RTT) and the periodic nature of the push cycle. With \texttt{FIREBASE\_INTERVAL\_MS} set to 1,000\,ms, the maximum theoretical delay from a physiological event to a database update is:
\begin{equation}
    T_{max} = T_{interval} + T_{RTT} + T_{proc} \approx 1000 + 300 + 5 = 1305\,\text{ms}
\end{equation}
The dashboard compensates for this by using a high-frequency WebSocket subscription to the RTDB, ensuring that once the data reaches the cloud, it is rendered on the clinician's screen in less than 100\,ms.

\subsection{FPS and Throughput Constraints}
The system operates at two distinct "clock" domains:
\begin{enumerate}
    \item \textbf{Telemetry Domain (1 Hz):} Constrained by the ESP32 upload task and the DHT22 recovery time.
    \item \textbf{Visual Intelligence Domain ($\approx$1.6 Hz):} Constrained by the 600\,ms capture-and-inference loop of the face recognition pipeline.
\end{enumerate}
This decoupled timing ensures that visual processing (which is computationally expensive) does not block the receipt or display of critical vital signs.

\subsection{Theoretical Model vs.\ Observed System Behaviour}

The latency model defined in Chapter~4 (Equation~\ref{eq:latency}) provides a framework
for expected end-to-end latency. Empirical validation is required to quantify RTT
and TLS overhead in the deployment environment.

The delta-threshold filter is expected to suppress redundant uploads during
steady-state monitoring (constant vitals). Quantifying the reduction in write
volume requires a controlled test that logs raw and filtered streams.

The AI forecasting model, based on OLS linear regression over a short window,
assumes near-linear short-term dynamics. This assumption is appropriate for
stable resting conditions but is expected to degrade during rapid transitions.
Future work should quantify this effect with controlled datasets.

\section{Error Analysis}
% ============================================================

\subsection{Embedded Subsystem}

Three categories of error were identified and analysed:

\textbf{Sensor read errors.}
The DHT22 can return NaN on the first read after power-up due to its required
warm-up delay. The NaN guard in \texttt{updateSensors()} discards invalid readings
before they can be uploaded.

\textbf{Records lost during outage (open).}
Records generated during a Wi-Fi outage are discarded.
The system does not implement local buffering or retry-on-reconnect.

\subsection{Backend, Web Dashboard, and AI Subsystems}

\textbf{Authentication expiry (open).}
Session cookies are signed and have a fixed expiration; a renewal mechanism is not
implemented in the current dashboard and should be added for long-lived sessions.

\textbf{Main-thread inference load (open).}
Face recognition runs on the main browser thread. Under heavy chart rendering or
low-power devices, this can lead to UI stutter; a dedicated Web Worker pipeline is
recommended for performance isolation.

\textbf{AI false positives (open).}
The Z-score model can flag short transient noise. A temporal persistence filter for
AI anomalies is not yet implemented and should be added to reduce alert noise.
